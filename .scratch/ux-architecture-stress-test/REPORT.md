## 1. Executive summary â€” the five things that most hurt UX

1. **Silent data-loss trap in the hashing core.** `scripts/body-hash.sh` returns the SHA of the *empty string* (`e3b0c442â€¦`) â€” exit code 0 â€” for any raw file missing its closing `---` frontmatter delimiter. The file is then marked "ingested" with that placeholder hash, idempotence treats it as unchanged forever, and the real body is **never processed and never surfaced as an error**. `verify-extract.sh` does not catch it either. This is the single highest-impact defect: it converts a trivial user typo into permanently-swallowed content with no feedback. (Critical/High.)

2. **The visual-output feature is broken on first use.** Neither the fresh skeleton nor the manifest creates a `diagrams/` directory (it's git-ignored), and neither `/wiki-query --visual` nor `/wiki-diagram` instructs the agent to `mkdir -p diagrams` before writing. Claude Code's Write tool *fails* on a missing parent dir, so the very first diagram a user requests dead-ends until the agent improvises. The headline output feature does not work out of the box. (High.)

3. **Two documented contracts contradict each other.** `wiki-lint.md` hard-requires "â‰Ą 2 `[[wiki-links]]`" in the Related section, while the `AGENTS.md` page template shows exactly **one**. An agent that faithfully follows the canonical template produces a page that the canonical linter immediately rejects, with no documented reconciliation. The schema fights itself. (High.)

4. **Self-contained prompts leak to AGENTS.md at the worst moments.** The operating commands repeatedly say "Read AGENTS.md (conventions)" with no scope or section pointer (it's ~275 lines), and the high-frequency `/wiki-ingest` steps 3â€“4 omit the full page frontmatter/section template entirely â€” forcing a context-switch on the most-used command. `/wiki-new` inlines its template; `/wiki-ingest` does not. The legibility model is inconsistent precisely where it's exercised most.

5. **Destructive factory action has no guardrail.** `/wiki-registry prune --apply` mutates `registry.jsonl` irreversibly with no confirmation, no diff of what's removed, and only an LLM-judgment gate ("only if the user explicitly asked"). Every other mutating command (`/wiki-lint --apply`) has an explicit "Apply these N fixes? [Y/n]" step. An over-eager agent reading "clean up my workspace" can delete entries silently. (High.)

Cross-cutting theme: the system is well-architected on the happy path but **fails quietly on the unhappy path** â€” malformed input, missing dirs, ambiguous prompts, and destructive actions all degrade silently instead of stopping with a clear message.

## 5. What's actually good (don't regress)

- **The factory cleanly separates leaf from factory:** generated wikis correctly do **not** ship `wiki-new`/`wiki-registry` (verified live). The "a generated wiki is a leaf, not a factory" invariant holds in the shell layer â€” the gap is only in *labeling*, not behavior.
- **The duplicate-name guard is well-engineered:** `new-wiki.sh` fails *before* scaffolding (registry check first) to avoid orphan skeletons, and the error even names the prune remedy. Right "fail fast, no partial state" pattern.
- **`render.sh` graceful degradation is excellent:** no renderer â†’ keeps the HTML, prints a real install hint, exits non-zero. "Degraded, not failed" is the right call for an optional feature.
- **Idempotence on the happy path works:** body-hash delta detection makes `/wiki-ingest` safely re-runnable on unchanged sources â€” the malformed-input case is the only crack.
- **`/wiki-lint --apply` and `wipe-meta-wiki.sh` model confirmation correctly** (explicit Y/n, `--yes` fallback). They're the template prune should copy.
- **`/wiki-diagram`'s scored candidate menu** is genuinely good UX â€” shows the 4 sub-scores and a one-line "why it fired," lets the user pick. The fix for #7 is to bring `/wiki-query --visual` up to this bar, not change `/wiki-diagram`.
- **`/wiki-new` inlines its seed-page template** â€” the right legibility pattern; `/wiki-ingest` should match it.
- **Verifiers exist and are honest:** `verify-*.sh`, `smoke-all.sh`, and the multi-wiki oracle give real coverage on Claude Code, and the README is candid that only Claude Code is harness-driven.
