You are executing `/wiki-registry $ARGUMENTS` from the `llm-wiki-bootstrap` **factory**. Your job is to show the local catalog of wikis you have generated, and (optionally) prune dangling entries. This is a thin wrapper over `scripts/registry.sh` — **do not reimplement** any registry parsing.

## Dispatch

Parse `$ARGUMENTS`:

- A `--workspace <dir>` flag, if present, is passed through. Otherwise the script defaults to `${LLM_WIKI_WORKSPACE:-$HOME/llm-wikis}`.
- If the remaining argument is `prune`, run prune; otherwise list.

### List (default)

```bash
scripts/registry.sh [--workspace <dir>] list
```

Surface the table verbatim. It shows each wiki's `NAME`, `SEEDED` (yes/pending), `STATUS` (`ok` / `MISSING` = registered but the directory is gone), and `DOMAIN`, plus any `UNREGISTERED` directories sitting in the workspace with no catalog entry.

After showing it, briefly interpret any drift:

- `MISSING` rows → the wiki dir was moved or deleted. Offer: "Run `/wiki-registry prune --apply` to drop dangling entries."
- `UNREGISTERED` lines → a wiki dir exists but was not created through the factory. (No automatic action — just note it.)

### Prune

```bash
scripts/registry.sh [--workspace <dir>] prune          # report only
scripts/registry.sh [--workspace <dir>] prune --apply  # remove dangling entries
```

Without `--apply`, prune only reports. Only pass `--apply` if the user explicitly asked to remove dangling entries — it edits `registry.jsonl` (it never deletes any wiki, only catalog lines).

## What you must NOT do

- Reimplement registry reading/parsing — always shell out to `scripts/registry.sh`.
- Edit any wiki, or `registry.jsonl` directly. Mutations go through `scripts/registry.sh` only.
- Pass `--apply` unless the user explicitly requested removal of dangling entries.
