llm-wiki-bootstrap · pitch interno

Sua wiki pessoal apodreceu porque manter dava mais trabalho que ler.

Isso muda quando o LLM é o bibliotecário. Você cura fontes e perguntas; ele faz o resto.

A premissa em 30 segundos

Tudo que faz wiki morrer — resumir, cross-linkar, achar contradição, atualizar índice — é exatamente onde LLM brilha e humano se cansa. Andrej Karpathy propôs inverter: o LLM mantém uma wiki persistente de markdown interlinkado, e o humano só joga fontes na pasta raw/ e faz perguntas.

Resultado: markdown puro, versionado em git, que fica mais inteligente a cada fonte ingerida. Sem vendor lock-in, sem app proprietário, sem embeddings opacos. Roda dentro do VSCode com qualquer extensão de AI.

O fluxo (5 passos, 5 minutos)

  1. git clone https://github.com/FrancyJGLisboa/llm-wiki-bootstrap meu-conhecimento
  2. code meu-conhecimento
  3. Apaga wiki/*.md se quer começar zerado. A wiki que vem é didática — é a wiki sobre wikis; serve de gabarito do que o padrão produz.
  4. Solta uma fonte em raw/: um PDF, um link, transcrito de YouTube, screenshot de slide.
  5. Abre o chat do seu AI no VSCode e diz: "Read AGENTS.md, then run the wiki-ingest workflow." O LLM escreve páginas em wiki/. A próxima pergunta — "what does this say about X?" — é respondida de dentro do projeto, sem reler os PDFs.

Qual extensão de AI no VSCode importa

Qualquer setup com loop agentic (lê arquivos, edita, executa terminal, segue regras) opera os 5 workflows. Conversacional puro também funciona, só exige que você dirija cada passo.

Os 4 problemas concretos que isso resolve

  1. "Releio os mesmos PDFs." A síntese fica escrita. Releitura vira consulta.
  2. "Meu Notion / Obsidian virou cemitério." Você não mantém. O LLM mantém.
  3. "ChatGPT esquece do que falamos semana passada." Conhecimento mora em arquivos .md, não em sessão. Cada sessão começa lendo a wiki que já existe.
  4. "5 fontes, 5 perguntas relacionadas, 5× o esforço." Wiki é interlinkada — uma pergunta atravessa muitas fontes via cross-refs já escritos.

O que NÃO prometo

  • Qualidade do output ≈ qualidade do prompt da sua extensão + qualidade das suas fontes. Lixo entra, lixo cross-linkado sai.
  • Runtime nunca foi testado em VSCode especificamente — só desenhado pra funcionar. Primeira pessoa que rodar /wiki-ingest em Cline ou Copilot é o smoke test (ver wiki/operation-ingest.md#verification-status).
  • Não substitui o trabalho de curar boas fontes e fazer boas perguntas. Essas continuam sendo as únicas duas coisas que continuam sendo trabalho seu.

Os três princípios que o pitch carrega implícitos

  1. File-over-app. Tudo é markdown em disco. Funciona em cat, grep, git, GitHub web, VSCode preview, sem dependência de viewer (Obsidian inclusive — opcional, não requerido).
  2. Yours. Os arquivos são seus. Você troca de LLM, de tool, de máquina; a wiki vai junto. Sem SaaS no caminho crítico.
  3. Bring your own AI. Qualquer LLM com tool-use (Claude, GPT, Gemini, modelo local via Ollama) opera o sistema. O projeto não amarra modelo.
