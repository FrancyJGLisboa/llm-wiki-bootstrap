Knowledge that compounds.

A simple way to make an LLM build and maintain a living knowledge base for your accounts, your sector, and your craft — so your prep stops starting from zero.

Based on the LLM knowledge-base pattern from Andrej Karpathy
01 · Where this came from
"using LLMs to build personal knowledge bases for various topics of research interest"
— Andrej Karpathy

Karpathy described spending less of his AI time manipulating code and more of it manipulating knowledge — stored as plain markdown and images. The idea spread fast among researchers. It transfers cleanly to a sales desk.

02 · The problem with how we use AI today

Nothing accumulates.

Today most people drop files into a chat and ask a question. The model retrieves a few chunks and answers. Fine for simple things — but every query rediscovers the knowledge from scratch.

→

No memory

The next question re-reads the same documents as if it had never seen them.

→

No cross-references

Connections between five separate sources get re-stitched on every single ask.

→

No compounding

Every good answer you got yesterday is thrown away. You pay for it again today.

03 · The flip

Build the wiki up front.

Instead of retrieving at query time, the LLM incrementally builds a persistent, interlinked wiki that sits between you and your raw sources. The cross-references already exist. Contradictions are already flagged. Knowledge compounds instead of being thrown away.

04 · The architecture — three layers

Clean and entirely yours.

Raw sources

Immutable

Articles, calls, emails, reports, news, data. The LLM reads them, never edits them. Your source of truth.

→
The wiki

LLM-owned

A folder of markdown files: summaries, account pages, concept pages, comparisons — all cross-linked and kept consistent.

→
The schema

The rules

A config file (an AGENTS.md) telling the LLM how the wiki is structured and which workflows to follow.

Mental model: the wiki is a codebase, /wiki-visualize is the viewer, the LLM is the writer, and the schema is the style guide.

05 · Three operations, that's it
INGEST

Drop in a source

Add a file to raw/ and tell the LLM to process it. It writes a summary, updates the index, and links it across every relevant existing page.

QUERY

Ask the wiki

Ask a question. It reads the relevant pages and answers — and a good answer can be filed back as a new page, so your explorations compound too.

LINT

Health-check it

Ask it to find contradictions, stale claims, orphan pages, and gaps worth a web search. The wiki stays clean as it grows.

06 · What happens when one source drops in

One file in. The whole wiki gets smarter.

1Read the raw source
5Flag any contradiction with existing claims
2Extract concepts, entities, claims, data points
6Update the index — the master catalog
3Write a summary page with tags + metadata
7Append to the log — timestamped change record
4Update existing pages with the new information
↻A single source can touch 10–15 pages
07 · Why it actually stays maintained

People abandon wikis because the upkeep grows faster than the value. The LLM doesn't get bored, doesn't forget a cross-reference, and edits 15 files in one pass. Maintenance cost drops to near zero.

    12 · Demo script — run this in order
    PlanAsk Claude (in plan mode) to design the wiki from the sample sources. It proposes the folders, the AGENTS.md schema, and a page layout.
    Build the scaffoldApprove. It creates raw/, wiki/, the schema, an index, and a log.
    IngestTell it to ingest the sources. Watch one file create account, person, and competitor pages — all cross-linked.
    VisualizeRun /wiki-visualize to render the graph of interlinked notes.
    Query"What are Acme's top objections and how have we handled them?" — answered from the wiki, no re-research.
    Backfill liveAsk something it doesn't have. It web-searches and files the answer back as a new page.
    LintRun the maintenance pass — contradictions, stale claims, orphan pages, gaps to fill.
